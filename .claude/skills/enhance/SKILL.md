# /enhance - 학습 문서 보강

## Triggers

- `/enhance` 호출 시
- 특정 docs 문서·카테고리의 보강을 요청할 때

## Arguments

`$ARGUMENTS` — 대상 문서 경로 또는 카테고리명 (생략 시 다음 작업 항목 제안)

---

## 원칙

- 정밀한 보강: 기존 기록의 맥락과 가치를 존중하며, 꼭 필요한 부분(모호한 표현, 누락된 핵심 원리)만 수정·추가. 불필요하게 전체를 다시 쓰지 않는다.
- 점진적 개선: 한 번에 완벽을 기하기보다, 학습이 깊어질 때마다 조금씩 업데이트.
- 내용 중심 우선순위: 중요도가 높은 내용을 먼저. 순서는 별도 논의.
- 편집 시 `/writing` skill의 컨벤션을 반드시 준수한다 (문체, 볼드 금지, 헤더 구성 등).

---

## 문서 개선 기준 (Done Criteria)

각 문서를 개선할 때 아래 기준을 충족해야 완료로 간주한다.

### 1. 설명 완결성

- "무엇인가(What)" 뿐 아니라 "왜 필요한가(Why)"와 "어떻게 동작하는가(How)"가 서술되어 있는가.
- 면접관이 "그게 왜 필요해요?"라고 물었을 때 문서 내용만으로 답변 가능한가.

### 2. 트레이드오프 명시

- 해당 기술/패턴의 장단점, 적용 조건, 대안이 명시되어 있는가.
- 예: 낙관적 락 vs 비관적 락, B-Tree 인덱스 vs Hash 인덱스, Look-aside vs Write-back.

### 3. 핵심 메커니즘의 구체성

- 추상적 서술 없이 내부 동작 단계가 명확하게 기술되어 있는가.
- 예: "MVCC로 격리 보장" → MVCC가 실제로 어떤 스냅샷을 어떻게 만들어 격리를 구현하는지.

### 4. 문장 품질

- 모호한 표현("잘 관리", "효율적으로", "이해함") 없이 기술 용어 중심으로 서술되어 있는가.
- 문장이 끊기거나 불완전하지 않은가.

## Process

하나의 문서(또는 문서 그룹)를 보강할 때 아래 5단계를 순서대로 진행한다.
단계별로 대화하며 진행. 한 번에 전체를 바꾸지 않고, 한 문서씩 집중한다.

### Step 1. Read — 현황 파악

대상 문서 전체를 읽는다.
문서가 여러 개인 경우 카테고리 내 파일 목록을 먼저 확인 후 관련 파일을 파악한다.

### Step 2. Audit — 기준 평가

Done Criteria 4가지로 현재 문서를 평가한다.
평가 결과를 항목별로 정리해 사용자에게 보여주고, 작업 범위를 합의한다.

| 기준 | 평가 항목 |
| :--- | :--- |
| 설명 완결성 | Why/How 없이 What만 서술된 항목이 있는가 |
| 트레이드오프 | 장단점·적용 조건·대안이 빠진 항목이 있는가 |
| 메커니즘 구체성 | "~를 통해", "~를 보장" 같은 추상 서술만 있는가 |
| 문장 품질 | 모호한 표현·끊긴 문장·불완전한 서술이 있는가 |

### Step 3. Plan — 작업 범위 결정

Audit 결과를 바탕으로 추가/수정/삭제 대상을 결정한다.
과도한 추가는 지양하고, 기준 미달 항목만 타깃으로 삼는다.
사용자 확인 후 진행.

### Step 4. Edit — 문서 보강

아래 3가지 관점을 순서대로 적용해 문서를 수정한다.

**4-1. Refine (문장·구조 다듬기)**

- 모호한 서술을 핵심 키워드 중심의 명확한 문장으로 교체
- 추상적 표현 → 정확한 기술 용어로 대체
- 예: "데이터를 잘 관리함" → "MVCC를 통한 트랜잭션 격리 수준 및 일관성 보장"

**4-2. Deepen (깊이 추가)**

- Why/How 보강: 무엇인지만이 아니라, 왜 필요한지·내부 동작 원리를 추가
- 트레이드오프: 해당 기술·패턴의 장단점·대안을 명시
- 메커니즘 구체화: 단계별 목록, 코드 예시, Mermaid 다이어그램으로 보완

**4-3. Internalize (실무·면접 관점 내재화)**

- 기술이 해결하는 문제(Problem)와 해결책(Solution)을 실무 맥락에서 서술
- "왜 이 기술인가?", "대안은 무엇인가?" 같은 심층 질문에 문장 내 논리로 대응 가능하도록

### Step 5. Verify — 검증

- 교차 검증: 공식 문서·신뢰도 높은 기술 서적과 대조해 오류 확인
- 최신성 확인: 학습 당시와 현재 기술 변화가 있다면 차이점 기록
